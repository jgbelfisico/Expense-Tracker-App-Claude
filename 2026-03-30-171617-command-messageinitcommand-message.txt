● Aquí está el plan completo:

  ---
  Estructura de archivos

  expense-tracker/
  ├── index.html   ← estructura y markup
  ├── style.css    ← presentación visual
  └── app.js       ← lógica, estado y DOM

  Tres archivos, separación clara de responsabilidades. app.js se carga al final
   del <body> para que el DOM esté listo.

  ---
  Modelo de datos

  Cada movimiento es un objeto JavaScript plano:

  {
    id:          string,           // crypto.randomUUID()
    type:        "income" | "expense",
    category:    string,           // "Alimentación", "Salario", etc.
    amount:      number,           // siempre positivo
    date:        string,           // "YYYY-MM-DD" (ISO 8601)
    description: string            // opcional, default ""
  }

  El estado de la app es un único array de estos objetos: movements = []. El
  signo está en type, no en amount — esto evita ambigüedades aritméticas.

  ---
  Cálculo de totales

  Se recalculan desde cero en cada render. El array es la única fuente de
  verdad:

  totalIncome   = suma de amount donde type === "income"
  totalExpenses = suma de amount donde type === "expense"
  balance       = totalIncome - totalExpenses

  Un solo recorrido del array. Como el dataset es pequeño (límite de
  localStorage), no hay problema de rendimiento.

  ---
  localStorage

  ┌───────────┬──────────────────────────────────────────────────────────────┐
  │  Aspecto  │                           Decisión                           │
  ├───────────┼──────────────────────────────────────────────────────────────┤
  │ Clave     │ "expense_tracker_movements"                                  │
  ├───────────┼──────────────────────────────────────────────────────────────┤
  │ Formato   │ JSON del array completo                                      │
  ├───────────┼──────────────────────────────────────────────────────────────┤
  │ Lectura   │ Al iniciar la app (con try/catch por si hay datos corruptos) │
  ├───────────┼──────────────────────────────────────────────────────────────┤
  │ Escritura │ Después de cada mutación (agregar o eliminar)                │
  └───────────┴──────────────────────────────────────────────────────────────┘

  Invariante: toda función que muta el array llama a saveToStorage() antes de
  retornar.

  ---
  Integración: interfaz, lógica y estado

  Flujo unidireccional:

  [ Evento DOM ] → [ Mutación de estado ] → [ Guardar en localStorage ] → [
  Render ]

  El DOM nunca tiene datos autoritativos — siempre es un reflejo del array.

  Funciones principales:

  ┌────────────────────┬─────────────────────────────────────────────────────┐
  │      Función       │                   Responsabilidad                   │
  ├────────────────────┼─────────────────────────────────────────────────────┤
  │ init()             │ Carga storage → render. Se llama una vez al cargar  │
  │                    │ la página                                           │
  ├────────────────────┼─────────────────────────────────────────────────────┤
  │ addMovement(data)  │ Valida → construye objeto → push → save → render    │
  ├────────────────────┼─────────────────────────────────────────────────────┤
  │ deleteMovement(id) │ Filter → save → render                              │
  ├────────────────────┼─────────────────────────────────────────────────────┤
  │ calculateSummary() │ Retorna { balance, totalIncome, totalExpenses }     │
  ├────────────────────┼─────────────────────────────────────────────────────┤
  │ render()           │ Llama a renderSummary() + renderMovementList() — un │
  │                    │  solo punto de entrada                              │
  └────────────────────┴─────────────────────────────────────────────────────┘

  Eventos:
  - form submit → leer campos, llamar addMovement(), resetear form
  - click en el contenedor de lista (delegado) → detectar botón de borrar →
  llamar deleteMovement(id)

  La delegación de eventos evita re-adjuntar listeners cada vez que se
  re-renderiza la lista.

  ---
  Decisiones clave

  ┌──────────────────────────┬──────────────────────────────────────────────┐
  │         Decisión         │                    Razón                     │
  ├──────────────────────────┼──────────────────────────────────────────────┤
  │ Derivar totales en cada  │ Evita bugs de sincronización entre estado    │
  │ render                   │ derivado y almacenado                        │
  ├──────────────────────────┼──────────────────────────────────────────────┤
  │ Un solo render()         │ Predecible, sin actualizaciones parciales    │
  ├──────────────────────────┼──────────────────────────────────────────────┤
  │ Delegación de eventos    │ No hay que gestionar listeners entre         │
  │ para borrar              │ re-renders                                   │
  ├──────────────────────────┼──────────────────────────────────────────────┤
  │ Amount siempre positivo  │ El signo vive en type, aritmética sin        │
  │                          │ ambigüedad                                   │
  └──────────────────────────┴──────────────────────────────────────────────┘

  ---
  ¿Apruebas este plan o quieres ajustar algo antes de empezar a escribir código?

● Plan de Implementación — Expense Tracker

  ---
  Fase 1 — Archivos base

  Objetivo: Tener el esqueleto mínimo funcional conectado.

  Archivos afectados: index.html, style.css, app.js

  Resultado: Una página en blanco que carga sin errores. El HTML tiene el <link>
   al CSS y el <script> al JS. La consola del navegador muestra un "App 
  iniciada" desde init().

  ---
  Fase 2 — Interfaz visual

  Objetivo: Construir todo el markup y estilos sin lógica. La UI existe pero no
  hace nada.

  Archivos afectados: index.html, style.css

  Resultado: Se ven tres secciones:
  1. Tarjetas de resumen (balance, ingresos, gastos) — con valores hardcodeados
  por ahora
  2. Formulario de agregar movimiento — con todos sus campos visibles
  3. Lista de movimientos — vacía pero con el contenedor listo

  La app se ve completa visualmente, incluyendo colores para ingreso/gasto.

  ---
  Fase 3 — Formulario de movimientos

  Objetivo: Capturar y validar los datos del formulario. Aún sin persistencia ni
   lista.

  Archivos afectados: app.js

  Resultado: Al enviar el formulario, addMovement() construye el objeto, lo hace
   console.log(), y resetea el form. Los campos inválidos muestran un error
  visible. El array movements crece en memoria pero no se muestra en pantalla
  todavía.

  ---
  Fase 4 — Lista de movimientos

  Objetivo: Renderizar el array movements en el DOM después de cada mutación.

  Archivos afectados: app.js, style.css (ajustes menores de filas)

  Resultado: Al agregar un movimiento aparece en la lista inmediatamente. Cada
  fila muestra tipo, categoría, monto, fecha y descripción. El botón de eliminar
   funciona (con delegación de eventos). La lista se re-renderiza limpiamente en
   cada cambio.

  ---
  Fase 5 — Cálculos

  Objetivo: Implementar calculateSummary() y conectarlo al DOM.

  Archivos afectados: app.js

  Resultado: Las tarjetas de resumen muestran valores reales derivados del
  array. Al agregar o eliminar un movimiento, balance/ingresos/gastos se
  actualizan al instante. El balance se pinta en verde si es positivo y en rojo
  si es negativo.

  ---
  Fase 6 — Persistencia con localStorage

  Objetivo: Que los datos sobrevivan al recargar la página.

  Archivos afectados: app.js

  Resultado: saveToStorage() escribe el array tras cada mutación.
  loadFromStorage() lo lee al iniciar. Al recargar la página, todos los
  movimientos siguen ahí. Un try/catch protege contra datos corruptos.

  ---
  Fase 7 — Filtros básicos

  Objetivo: Permitir filtrar la lista por tipo y/o categoría.

  Archivos afectados: index.html (controles de filtro), app.js (lógica de
  filtrado)

  Resultado: El usuario puede filtrar por tipo (todos / ingresos / gastos) y por
   categoría. La lista se actualiza sin tocar el array original — el filtro
  actúa solo sobre la vista. El resumen sigue calculándose sobre todos los
  movimientos.

  ---
  Fase 8 — Revisión final

  Objetivo: Pulir, limpiar y verificar que todo funciona en conjunto.

  Archivos afectados: Los tres

  Resultado:
  - Sin errores en consola
  - La app funciona tras recargar, con datos persistidos
  - El código en app.js está ordenado (primero estado, luego funciones, luego
  eventos, luego init())
  - El CSS es consistente y la UI se ve limpia en mobile y desktop
  - Se actualiza CLAUDE.md con los comandos reales y la arquitectura final

● Fase 3 completa. El flujo completo al enviar el formulario:

  submit → handleFormSubmit → validate → showError / addMovement → form.reset()

  ---
  Qué hace cada función:

  generateId() — Usa Date.now() para crear un ID numérico único como string.
  Simple y suficiente para datos locales.

  validate(type, category, amount, date) — Revisa los cuatro campos obligatorios
   en orden. Devuelve el primer mensaje de error que encuentra, o null si todo
  está bien. Las reglas del monto son tres: que exista, que sea un número, y que
   sea mayor a cero.

  showError / hideError — Escriben o borran el texto del #form-error y alternan
  la clase hidden (que ya está definida en el CSS con display: none).

● Cómo funciona el Expense Tracker

  ---
  Un movimiento es solo un objeto JavaScript

  Cuando agregas un gasto o ingreso, el programa crea un objeto — básicamente
  una "ficha" con toda la información:

  {
    id:          "1743350000000",   // número único para identificarlo
    type:        "expense",         // ¿es ingreso o gasto?
    category:    "Alimentación",
    amount:      120,               // siempre positivo
    date:        "2026-03-30",
    description: "Supermercado"
  }

  Todos esos objetos viven dentro de un array llamado AppState.movements.
  Imagínalo como una lista de fichas apiladas:

  movements = [ ficha1, ficha2, ficha3, ... ]

  Eso es todo el "cerebro" de la app. Si el array está bien, todo lo demás se
  puede calcular.

  ---
  Guardar datos: el problema y la solución

  El problema: JavaScript vive en la memoria del navegador. Si cierras la
  pestaña, todo desaparece.

  La solución: localStorage — una pequeña caja de texto que el navegador guarda
  en tu computadora entre sesiones.

  Pero localStorage solo acepta texto, y el array es un objeto JavaScript. Por
  eso hay una conversión en dos pasos:

  Guardar:   array  →  JSON.stringify()  →  texto  →  localStorage
  Recuperar: localStorage  →  texto  →  JSON.parse()  →  array

  JSON.stringify convierte el array en un string así:
  '[{"id":"123","type":"expense","amount":120,...}]'

  Y JSON.parse lo convierte de vuelta al array original.

  Esto ocurre automáticamente cada vez que agregas o eliminas un movimiento.

  ---
  Calcular totales: un solo recorrido

  La función calculateSummary() recorre el array una vez de principio a fin,
  como leer una lista de arriba a abajo:

  Para cada movimiento:
    ¿Es "income"?  → súmalo a totalIncome
    ¿Es "expense"? → súmalo a totalExpenses

  Al final:
    balance = totalIncome - totalExpenses

  En código, eso se ve así:

  let totalIncome   = 0;
  let totalExpenses = 0;

  AppState.movements.forEach(function(movement) {
    if (movement.type === "income") {
      totalIncome += movement.amount;
    } else {
      totalExpenses += movement.amount;
    }
  });

  Cada vez que agregas o eliminas un movimiento, esta función se ejecuta de
  cero. No guarda resultados intermedios — recalcula todo desde el array. Eso
  garantiza que los números siempre sean correctos.

  ---
  El filtro: no toca el array original

  Cuando seleccionas "Gastos", el programa no borra los ingresos del array. Solo
   cambia una variable:

  AppState.filter = "expense"  // antes era "all"

  Luego, al pintar la lista, crea una copia filtrada:

  const visible = AppState.movements.filter(function(m) {
    if (AppState.filter === "all") return true;
    return m.type === AppState.filter;
  });

  visible es un array temporal que existe solo para mostrar en pantalla. El
  array real (AppState.movements) nunca cambia. Por eso el resumen de balance,
  ingresos y gastos no se ve afectado por el filtro — sigue calculándose sobre
  todos los movimientos.

  Cuando vuelves a "Todos", AppState.filter vuelve a "all" y la lista muestra
  todo otra vez.

  ---
  Cómo se actualiza la interfaz

  La interfaz nunca "recuerda" datos por su cuenta. Es solo un espejo del array.

  El flujo siempre sigue este orden:

  1. El usuario hace algo  (llenar el form, hacer clic en eliminar)
          ↓
  2. El array cambia       (se agrega o se borra un objeto)
          ↓
  3. Se guarda en storage  (saveToStorage)
          ↓
  4. Se repinta el DOM     (renderMovementList + renderSummary)

  "Repintar" significa vaciar el contenedor de la lista con innerHTML = "" y
  volver a construir cada fila desde cero leyendo el array. Es como borrar una
  pizarra y volver a escribir todo.

  Esto parece ineficiente, pero para una lista de movimientos personales es
  perfectamente rápido y mucho más simple de entender que actualizar solo las
  partes que cambiaron.

  ---
  La idea central

  Todo el proyecto gira alrededor de una regla:

  ▎ El array es la única fuente de verdad. La interfaz es solo su reflejo.

  Si quieres saber cuánto dinero tienes → miras el array.
  Si quieres mostrar algo en pantalla → lees el array y lo pitas.
  Si quieres cambiar algo → modificas el array y repintas.

  Nunca al revés.

  ---
  Qué se implementó

  1. Categorías por tipo
  El <select> arranca vacío. renderCategoryOptions(type) lo llena dinámicamente
  con las 5 categorías de ingresos o las 8 de gastos según el radio
  seleccionado. Al cambiar el tipo en el formulario, la lista se actualiza al

  correcto y se preselecciona la original.

  2. Gráfica de barras
  Dibujada con Canvas 2D, sin librerías. renderChart() ajusta canvas.width al
  ancho CSS real antes de dibujar (esto evita que las barras se vean borrosas en
   pantallas retina). Dos barras proporcionales al máximo entre ambos valores,
  con el monto encima y la etiqueta abajo. Se actualiza en cada mutación.

  3. Editar movimiento
  Cada fila tiene un botón ✎ (btn--edit). Al hacer clic: el formulario se
  rellena con los datos del movimiento, el título cambia a "Editar movimiento",
  el botón de enviar dice "Guardar cambios" y aparece "Cancelar". La fila en
  edición se resalta con un borde azul (movement--editing). saveEdit() reemplaza
   el objeto en el array conservando el mismo id. Cancelar o guardar devuelve el
   formulario a modo agregar.

  renderMovementList() ahora ordena por fecha (no por inserción) y agrupa en un
  Map por "YYYY-MM". Por cada mes pinta un encabezado con el nombre del mes y
  sus totales de ingresos/gastos en verde y rojo. Los movimientos aparecen
  debajo de su mes correspondiente.

● El código está limpio y correcto. Resumen de lo que se hizo y por qué:

  ---
  Problemas encontrados y correcciones

  1. Dos funciones de cálculo idénticas → una sola summarize(array)

  calculateSummary() y calcMonthSummary() hacían exactamente el mismo loop con
  nombres de variables distintos. Se unificaron en summarize(movements) que
  recibe cualquier array y devuelve { income, expenses, balance }. Ahora se usa
  en tres lugares: resumen global, gráfica y encabezados de mes.

  2. Bloque de 4 líneas repetido en 3 funciones → afterMutation()

  addMovement, deleteMovement y saveEdit terminaban con el mismo bloque:
  renderMovementList();
  renderSummary();
  renderChart();
  saveToStorage();
  Extraído en afterMutation(). Si mañana se agrega un render nuevo, se cambia en
   un solo lugar.

  3. calculateSummary() ejecutada dos veces por mutación

  renderSummary y renderChart llamaban cada una a calculateSummary() por
  separado. Ahora afterMutation() llama a summarize() una vez y pasa el
  resultado a ambas funciones como argumento. El mismo patrón se aplica en
  init().

  4. Alineación de validate()

  El código usaba espacios en blanco extremos para alinear los return, lo que
  dificultaba la lectura. Se usa una alineación moderada y consistente.

  ---
  No se cambió nada de lógica, comportamiento, HTML ni CSS. La app funciona
  exactamente igual.
